Ancestors

COMMIT ID followed by

  • ^ OR ^1 - first parent of the commit
  • ~ OR ~1 - first parent of the commit
  • ^2 - second parent of the commit (only merge commit)
  • ~2 - the first parent of the first parent - the grandparent
  • ~2^2 - second parent of the grandparent (only merge commit)

branch into which we merged.

Jak rebasować, żeby nie bolało

Rebase to świetne narzędzie, ale...

  • zakłamauje historię
  • może wywołać konflikty których by nie było przy mergu
    • a w związku z tym, może wprowadzić błąd na produkcję
    • pomijając, że to dokłada dodatkowej roboty

Demo

Przygotowanie przed warsztatatami

  • git clone https://github.com/git-warsztaty/silverflow-rebase
  • cd silverflow-rebase
  • git reset --hard HEAD^
  • git checkout simplify_checking_if_person_is_old_enough_to_rent_a_car

Pokazujemy

  • gitk - Chcemy dołączyć feature_brancha, pokazujemy commity od mastera
  • gitk --all - Sprawdzamy, że na masterze doszedł tylko jeden commit
  • git rebase origin/master - Przed dołączaniem rebasujemy, żeby historia była prosta
  • git mergetool ; git rebase --continue - N razy
  • gitk - Historia wygląda tak jak chcieliśmy; Było kilka konlfitów, a sytuacja w tym demo jest bardzo prosta, życie zwykle nie jest takie proste; Konflikty trudne, zwłaszcza jak jakieś przeniesienia kodu. Kto wie ile było by konfliktów, gdybyśmy robili merge?
  • git reset --hard origin/simplify_checking_if_person_is_old_enough_to_rent_a_car - Cofamy się w czasie przed rebase
  • gitk --all - Jesteśmy cofnięci w czasie
  • git merge origin/master - Zamiast rebase robimy merge
  • git mergetool
  • git status ; git commit - All conflicts fixed
  • gitk - Był tylko jeden konflikt, ale historia właśnie nam się zagmatwała. Jest jeszcze gorzej. Z rzadka zdarzy się tak, że przy rebase mamy N konfliktów, a przy merge ani jednego. Ostateczna wersja na masterze może nie konfliktować z ostateczna z feature brancha.
  • git reset --hard origin/simplify_checking_if_person_is_old_enough_to_rent_a_car - Znowu cofamy się w czasie
  • gitk --all - No to jak rebasować, żeby nie bolało
  • Najpierw zaczynamy rebase tradycyjnie. Jeśli nie ma konfliktów no to nie ma problemu. Jeżeli zaczynają się konflikty to przerywamy przez git rebase --abort
  • Kto wie jak zrobić liniowego merga? Merge z flagą squash działa tak, jakbyśmy całego naszego fetaure_brancha zrebasowali do jednego commita, następnie przerzucili go cherry-pickiem
  • git merge origin/master --squash - Robimy merga, ale w wersji squash, czyli liniowo
  • git mergetool ; git commit - Rozwiązujemy konflikty jak przy zwykłym mergu
  • gitk --all - Historia dalej liniowa i już mająca zmianę z mastera
  • git rebase origin/master - Rebasujemy
  • git checkout --theirs . - Rozwiązujemy konflikty automatycznie, wybierając nasze zmiany
  • git add . - Dodajemy rozwiązane konflikty
  • git rebase --continue - Kontynuujemy ...
  • gitk - I mamy to!
  • Kto spróbuje wytłumaczyć co tu się właśnie wydarzyło.

Ostrzenie konsoli

Obawy przed konsolą

  • Pewnie trzeba się strasznie opisać, a to będzie wolno szło
  • Pewnie będzie problem spamiętać komendy
  • To jest jakiś niewygodny przeżytek

Kiedy nie warto kurczowo trzymać się konsoli?

  • Tool bywa lepszy do diffa
  • Toole ładniej wizualizują historię branchów
  • W dobrym IDE wygodniej się patrzy na Git Blame
  • Nie warto rozwiązywać konfliktów bez toola
Oczywiście, nie warto być fanatykiem i jak coś jest wygodniejsze w toolu, to lepiej wykorzystać toola. Tylko, że gdy się ma wydajnie ogarniętą konsolę to jest rzadkość. Ale np. git blame zawsze robię z Ide. Przeszukiwanie historii czasami z IDE lub gitk, ale tylko gdy nie mam pomysłu czego szukam, w przeciwnym wypadku log ogarnia mi to bardzo dobrze. Merge robię zawsze z kdiff3. Diffy oglądam zawsze z kdiff3 lub vimdiff -> ten drugi działa szybciej i nie odrywa mnie od konsoli.

Myszka jest do eksplorowania

Są toole z dobrze ogarniętymi
skrótami klawiszowymi

Np. Rider jest dobrze ogarnięty skrótowo

Ale dobrze ogarnięta konsola wygrywa

  • Z dobrze ogarniętą konsolą pracuje się wydajniej
  • Konsola zawsze ma więcej możliwości - żaden tool nie pokryje wszystkich casów
  • Konsola jest uniwersalna
  • Nie ma toola działającego szybciej niż konsola
  • Konsola zachęca do lepszego zrozumienia Gita
  • Toole często zachęcają do złych praktyk i budują złe nawyki
  • Toole często zachęcają do klikania zamiast używania skrótów klawiszowych
  • Wiele (popularnych) tooli działa wolno i odrywa od flow
Przykłady złych praktyk: długi commit title, korzystanie z jakichś magic sync buttonów i innych takich złych rozwiązań, przycisk Git Flow.

Co to znaczy dobrze ogarnięta konsola?

  • Konsola zawsze pod ręką
    • Zdecydowanie najlepiej gdy jest zintegrowana w IDE
    • Opcjonalnie wyskakująca od skrótu klawiszowego
      (choć to mało przy otwartych na raz dwóch lub więcej projektach)
  • Konsola z wydajnym autouzupełnianiem
  • Zdecydowanie dobrze, gdy konsola ma wydajny
    reverse search
  • Ustawione Git aliasy i aliasy na poziomie shella
  • Edytor konsolowy lub przynajmniej szybko otwierający się

reprezentowany za pomocą hasha SHA-1
Dzięki temu, można wskazać dowolny commit i podejrzeć cały stan systemu z ówczas, a nie tylko jakie diffy ten commit ze sobą niósł.

Wiedząc jaka jest wersja na produkcji, zawsze można wrócić do niej w kodzie i zrobić drobną poprawkę bez wprowadzania na produkcję kolejnych commitów.

Credentials helper

Każdy push, fetch (czyli również pull), wymaga autoryzacji...

Wbudowane credentials helpery

  • cache
    • git config --global credential.helper 'cache --timeout=300'
      Bezpieczny, ale tylko tymczasowy (in memory).
      Opcjonalnie własny timeout w sekundach (domyślny 900 - 15minut).
  • store
    • git config --global credential.helper store
      Wygodny, ale niebezpieczny - credentiale zapisane w pliku tekstowym niezaszyfrowane.

libsecret - wygodny i bezpieczny credentials helper dla Linux

Przykładowa instalacja dla Linux Mint / Ubuntu
  1. sudo apt-get install libsecret-1-0 libsecret-1-dev
  2. cd /usr/share/doc/git/contrib/credential/libsecret
  3. sudo make

Konfiguracja (gdy już zainstalowany)
  • git config --global credential.helper /usr/share/doc/git/contrib/credential/libsecret/git-credential-libsecret
Szerszy opis na blogu gitwarsztaty.pl
